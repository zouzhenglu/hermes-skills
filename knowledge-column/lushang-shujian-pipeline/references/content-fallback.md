# 书籍内容获取策略

## 优先级链

按此顺序尝试获取书籍内容：

### 1. 用户提供的电子书文件（最佳）

```
~/knowledge-column/book-pool/<filename>.pdf
~/knowledge-column/book-pool/<filename>.epub
~/knowledge-column/book-pool/<filename>.txt
```

提取命令：
```bash
pip3 install --break-system-packages pymupdf
python3 -c "import pymupdf; doc=pymupdf.open('文件路径'); [print(p.get_text()) for p in doc]"
```

### 2. Web 搜索（次选）

搜索关键词模式：`书名 核心观点` / `书名 目录` / `书名 书评`

### 3. Agent 内置知识（畅销书降级方案）

全球销量 > 100 万册且在中文世界广泛传播的畅销书，Agent 训练数据覆盖度通常足够产出核心内容。产出时**必须告知用户来源**。

不适用：小众书、新书（2024+）、中文原创未广泛传播的。

### 4. 要求用户补充（兜底）

标记 `status: needs_content`，请用户提供文件。

## 诚实原则

| 来源 | 告知用户用语 |
|------|------------|
| 电子书原文 | 「基于你提供的电子书拆解」 |
| Web 搜索 | 「基于公开书评和笔记整理，细节可能不完整」 |
| 内置知识 | 「没拿到电子书，内容来自训练数据。核心框架应该对，建议对照原文确认细节」 |

**不要假装读过一本你没有原文的书。**

## 拿到电子书后的验证流程（重要）

当电子书在初版内容产出**之后**才到位时，必须执行以下验证：

1. **提取全文** → pymupdf 生成 txt
2. **验证核心定义**：搜索关键词，核对术语是否准确
   - 例：脚本写「舒服」→ 原书是「愉快」→ 必须修正（含义不同）
3. **验证书内结构**：目录 vs 已生成的内容规划
   - 例：自编的「七层堆积」框架 → 原书实际是「无法放手的三类人」→ 替换
4. **验证书内案例**：搜索「体验谈」「比如」等关键词
   - 编造的故事可以保留但标注「走正路自己的经历」
   - **书里有更好的真实案例 → 必须替换为原文**
5. **验证书内金句**：用原文引用替代凭记忆写的「大意是」
   - 原文：「进，则出。出，则进。然后，再出。」
   - 而不是：「山下英子大概说新陈代谢很重要」

**验证成本**：5-10 分钟 grep + 逐条比对。远比事后被读者指出错误代价小。
