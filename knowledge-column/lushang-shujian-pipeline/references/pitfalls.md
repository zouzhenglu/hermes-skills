# 本期踩过的坑 & 教训

## 语音选型

- ❌ `zh-CN-YunyangNeural` +10% 语速：像新闻播报，开头太赶。「林间」原名用这个，用户反馈不好。
- ✅ `zh-CN-YunxiNeural` +0% 语速：讲故事风格，温暖自然，自带叙述节奏。改名「走正路」后切换。
- 教训：讲书类音频需要「人味」，不能赶。0% 语速是最优解，不要预设「通勤路上要提神就加速」。

## 人设迭代

- 「林间」：温暖朋友型，轻快、自嘲。太像「另一个 AI」，不够有辨识度。
- 「走正路」：通透叙事型，沉稳、咀嚼过再讲。用户主动要求改名，说明需要更有分量的人设。
- 教训：第一个人设不要追求「亲切」，要追求「可信」。亲切可以演，可信需要料。

## 文章定位

- 初版文章 = 音频的文字版（七个洞察、讲故事）。用户反馈：音频讲过的东西文章不要重复。
- 修正：文章做音频做不到的事——表格、清单、框架、对比。
- 教训：每次生成文章前问自己：「这段内容，音频能讲吗？」能讲的就一笔带过，不能讲的展开。

## 周一引读

- 初版周一只是预告（「这周要讲断舍离」）。问题：浪费了用户一天的通勤时间。
- 修正：周一 = 引读（3min）+ 全书最颠覆的一个观点（5min）。让周一有独立价值。
- 教训：每天的内容必须有独立收获。用户可能只听了这一天。

## 书池 vs 推荐

- 延伸阅读（周日推荐的三本书）不等于下一本要拆的书。
- 池子 = 用户选、用户给书、深度拆解。推荐 = Agent 推、顺带一提、听众反馈后决定。
- 教训：不要自作主张把推荐书当下一本书处理。

## 图片生成

- image_generate 需要 FAL_KEY 环境变量。没有就报错，不是 Agent 的问题。
- 告知用户去 fal.ai 免费注册即可。

## 文本显示

- Markdown 原文太素，用户看不下。需要生成 WeChat 兼容的 HTML 富文本版。
- 关键设计元素：卡片分层、表格着色、行动清单左边框、金句深色卡片。
- 模板保存为 `templates/article_wechat.html`。

## 内容准确性：无电子书 vs 有电子书（新增 ⚠️）

- **最大的坑**：凭记忆/训练数据写的脚本，拿到电子书后发现多处不准确。
  - 「舒服」→ 原书是「愉快」（含义不同：舒服是被动的，愉快是主动的）
  - 自己编的「七层堆积」→ 原书没有这个框架，实际是「无法放手的三类人」
  - 编造的「茶具故事」→ 原书有真实「体验谈」（洋子小姐、前夫家具等）
- **教训**：
  1. 无电子书时，声明「内容基于训练数据，拿到原文后会核对」
  2. 拿到电子书后，**必须**搜索关键词逐条验证：核心定义、章节结构、原文案例
  3. 编造的故事不丢人——但如果原书有更好的真实案例，必须替换
  4. 术语错误（舒服→愉快）是最容易被读者发现的硬伤

## 公众号发布

- Hermes 的 WEIXIN_* 环境变量用于消息平台（聊天 bot），不是公众号 API。
- 公众号草稿/发布需要独立的 WECHAT_APP_ID 和 WECHAT_APP_SECRET。
- 参见 `references/wechat-publish.md`。

**新增踩坑（Session 2026-07-12）：**

1. **JSON 编码致命陷阱**：`requests.post(json=payload)` 会把中文转成 `\uXXXX` 导致草稿全部乱码。必须 `json.dumps(ensure_ascii=False).encode('utf-8')` + `headers={'Content-Type': 'application/json; charset=utf-8'}`。详见 `references/wechat-api-notes.md`。

2. **标题硬限制**：个人订阅号 ≤8 个中文字符（24 字节 UTF-8），超长报 45003。后台发布时可手动加长到 64 字。

3. **author/digest 字段**：个人订阅号不能用。`author` 报 45110，`digest` 不稳定。

4. **封面图尺寸**：不能用 1×1 像素图（报 53402）。用 Pillow 生成 900×500 即可。

5. **API 无法设原创/AI标注/合集/赞赏**：个人订阅号只能后台手动。**不要在文章 HTML 里用文字假装这些功能**——文章是成品，不是说明书。底部只需栏目署名 + 播客入口 + 点赞引导。

6. **WeChat HTML 兼容性**：实测 `linear-gradient`、`box-shadow`、`flex`、`border-radius` 在微信内置浏览器和草稿预览中均正常渲染。但**能用 ≠ 随便用**：
   - 渐变两色跨度要够大（`#5b7a6f→#6b8e80` 肉眼分不出，不如纯色）
   - border-radius 要按元素类型选值（卡片可 8px，小元素保持 4px）
   - 改字号必须同步调 line-height
   - **禁止正则全局替换 CSS 值**，逐元素判断

7. **CSS 机械替换是陷阱**（新增 2026-07-13）：
   - 用脚本全局 `border-radius:4px→8px` 会把顶部音频提示条也改大，不协调
   - 全局 `font-size:14→15px` 不改 line-height，行距变挤
   - `linear-gradient(135deg,#5b7a6f,#6b8e80)` 两色几乎一样，纯色反而更干净
   - 教训：排版优化必须逐元素判断，脚本批量替换必翻车。已有样式如果看着不差就别动

7. **音频文件命名**：必须包含书名，如 `book-001_断舍离_ep01_引读.mp3`。本地好找，上传微信素材库后也能一眼看到书名。

8. **文章元说明文字不要放**：用户反馈「阅读指南」「这不是音频的文字版」多余，直接删掉。文章不需要解释自己是什么。

9. **SVG 字体**：Linux 服务器渲染 SVG 时，`font-family` 不能写 `'PingFang SC'` 或 `'Microsoft YaHei'`——这些字体不存在导致中文变豆腐块（□），且为专有字体不可嵌入分发。必须用 `'Noto Sans CJK SC'`（SIL OFL），安装：`sudo apt-get install fonts-noto-cjk`。

10. **阅读进度条提取标题**：微信文章底部自动显示阅读进度条，抓取第一个醒目标题做标签。如果标题含「30 秒看懂」，进度条显示为「全书骨架 · 30 秒」——读者以为底部多了一段文字。解决：标题用纯描述（如「全书骨架」），时间/数量提示放正文。

11. **音频嵌入文章**：`mpvoice` 标签被微信 API 拦截（`invalid content`）。语音素材可上传但无法 API 嵌入，只能后台编辑器手动插入。分集文章只需顶部放音频占位提示 + 语音 ID。
